# Dependency Injection Reference — Audit Guide

Use this file to evaluate whether the audited project uses dependency injection correctly, and to calibrate the severity of DI-related findings.

---

## Expected DI Setup

A well-configured Flutter project uses `get_it` + `injectable`:

```yaml
# pubspec.yaml — expected entries
dependencies:
  get_it: ^8.x.x
  injectable: ^2.x.x
dev_dependencies:
  injectable_generator: ^2.x.x
  build_runner: ^2.x.x
```

Expected files:
- `lib/core/di/injection.dart` — `GetIt.instance` + `@InjectableInit`
- `lib/core/di/injection.config.dart` — generated by `build_runner`

---

## Annotation Map — Who Gets What

| Type | Correct annotation | Lifetime |
|---|---|---|
| Bloc, Cubit | `@injectable` | Transient — new instance per `BlocProvider` |
| Use case | `@injectable` | Transient |
| Repository impl | `@LazySingleton(as: Repository)` | Singleton, bound to interface |
| Datasource impl | `@LazySingleton(as: Datasource)` | Singleton, bound to interface |
| Router | `@lazySingleton` | Singleton |
| Firebase services | `@module` + `@lazySingleton` | Singleton via module |
| Services (auth, analytics) | `@lazySingleton` | Singleton |

---

## Correct Registration Patterns

### Repository bound to its interface
```dart
@LazySingleton(as: ProductRepository)
class ProductRepositoryImpl implements ProductRepository { ... }
```
**Audit signal:** If `@injectable` (not `@LazySingleton(as: ...)`) is used on a repository impl, the interface binding is missing and code likely depends on the concrete class.

### module for 3rd-party instances
```dart
@module
abstract class FirebaseModule {
  @lazySingleton
  FirebaseFirestore get firestore => FirebaseFirestore.instance;
}
```
**Audit signal:** If `FirebaseFirestore.instance` is called directly inside repositories or datasources (not injected), DI is being bypassed.

## DI Absence — Severity Assessment

| Observation | Severity |
|---|---|
| `get_it` + `injectable` present and used consistently | ✅ No issue |
| `get_it` present but `ServiceLocator.get<X>()` called inside domain layer | HIGH |
| Dependencies instantiated with `MyClass()` inside constructors | HIGH |
| `GetIt` used as service locator in widget `build()` without Bloc intermediary | MEDIUM |
| No DI at all — all dependencies created inline or via static singletons | HIGH |
| `Provider` used only for app-level config (theme, locale) alongside `get_it` for DI | LOW / Acceptable |

---

## Service Locator Anti-Pattern

Using `getIt<X>()` directly inside `domain/` or `data/` bypasses inversion of control:

```
✅ Correct: GetIt only called in presentation/screen/ (BlocProvider.create)
✅ Correct: GetIt called in app/ entry point (AppRouter, configureDependencies)

❌ Violation: getIt<ProductRepository>() called inside a use case
❌ Violation: getIt<FirebaseFirestore>() called inside a domain entity
❌ Violation: getIt<MyBloc>() called inside another Bloc
```

## Build Runner Signals

The presence of `injection.config.dart` confirms `build_runner` was executed. Its absence means:
- Either DI is not set up
- Or generated file is gitignored (check `.gitignore`)

Check `.gitignore` for `*.config.dart` — if present, DI may still be configured but generated files are excluded from the repo (common in some teams).

## Scoped DI — Advanced Pattern

For features that should clean up their registrations on exit (e.g., checkout flow):

```dart
// Create scope on feature entry
getIt.pushNewScope(scopeName: 'checkout');

// Drop scope on feature exit
getIt.dropScope('checkout');
```

**Audit note:** Absence of scoped DI is not a violation — it's an advanced optimization. Only flag if the project shows memory/lifecycle issues correlated with feature Blocs not being disposed.

## Testability Signal

DI is directly correlated with testability. A well-injected codebase allows:

```
✅ Mockable: class ProductBloc(this._getProducts) — constructor injection
❌ Not mockable: class ProductBloc() { final _uc = GetProductsUseCase(); }
```

If `test/` contains `mock_*.dart` or `*_mock.dart` files generated with `mocktail` or `mockito`, DI is likely correct. Absence of tests that mock dependencies is a maintainability signal.
