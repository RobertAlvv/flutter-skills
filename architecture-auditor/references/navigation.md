# Navigation Reference — Audit Guide

Use this file to evaluate whether the audited project uses a scalable, centralized routing strategy.

---

## Expected Navigation Setup

A well-configured Flutter project uses `auto_route` with a centralized router:

```yaml
# pubspec.yaml — expected entries
dependencies:
  auto_route: ^10.x.x
dev_dependencies:
  auto_route_generator: ^10.x.x
  build_runner: ^2.x.x
```

Expected files:
- `lib/core/router/app_router.dart` — `@AutoRouterConfig` + all routes declared
- `lib/core/router/app_router.gr.dart` — generated by `build_runner`
- `lib/core/router/guards/auth_guard.dart` — route guards (if auth exists)

---

## Correct Routing Structure

### Centralized router
All routes declared in one place. No `Navigator.push` scattered across the codebase:

```dart
@lazySingleton
@AutoRouterConfig(replaceInRouteName: 'Page,Route')
class AppRouter extends RootStackRouter {
  @override
  List<AutoRoute> get routes => [
    AutoRoute(page: SplashRoute.page, initial: true),
    AutoRoute(page: LoginRoute.page),
    AutoRoute(
      page: ShellRoute.page,           // Bottom nav shell
      children: [
        AutoRoute(page: HomeRoute.page),
        AutoRoute(page: SearchRoute.page),
        AutoRoute(page: ProfileRoute.page),
      ],
    ),
  ];
}
```

### Screen registration
Every screen that is a route must be annotated:

```dart
@RoutePage()
class ProductScreen extends StatelessWidget { ... }
```

**Audit signal:** Screens without `@RoutePage()` that appear in navigation flow means routing is partially manual.

---

## Routing Violation Severity

| Observation | Severity |
|---|---|
| Centralized `AppRouter` with all routes, `@RoutePage()` on all screens | ✅ No issue |
| `auto_route` or `go_router` used consistently | ✅ No issue |
| `Navigator.push(context, MaterialPageRoute(...))` in one or two places | LOW |
| `Navigator.push` used throughout — no centralized router | HIGH |
| Routes defined in multiple disconnected files | MEDIUM |
| Navigation called directly from a Bloc (not via `BlocListener`) | MEDIUM |
| Route arguments passed as full objects instead of IDs | LOW |

---

## Correct Navigation Call Patterns

Navigation must happen in the **presentation layer** only, triggered by `BlocListener`:

```
✅ Correct:
BlocListener<AuthBloc, AuthState>(
  listener: (context, state) {
    state.whenOrNull(
      authenticated: () => context.router.replaceAll([const HomeRoute()]),
    );
  },
)

❌ Violation: Bloc calls context.router directly (requires BuildContext in Bloc)
❌ Violation: Navigation in BlocBuilder.builder (side effect mixed with UI rendering)
❌ Violation: Navigator.push inside a use case or repository
```

---

## Route Arguments — Passing Data

Only primitives (IDs, strings, enums) should be passed as route arguments. Never full entities:

```
✅ Correct: context.router.push(ProductDetailRoute(productId: product.id))
❌ Violation: context.router.push(ProductDetailRoute(product: product))
```

Passing full objects breaks deep linking, serialization, and route restoration.

---

## Tab Navigation

For apps with bottom navigation, `AutoTabsScaffold` or `AutoTabsRouter` manages tab state:

```
✅ Correct: Shell route with AutoTabsScaffold — tab state preserved
❌ Violation: Manually managing tab index with setState in a StatefulWidget
❌ Violation: Each tab rebuilds completely on switch (no preservation)
```

---

## Route Guards

Auth-gated routes must use declarative guards, not inline checks inside screen `build()`:

```
✅ Correct:
AutoRoute(page: ProfileRoute.page, guards: [AuthGuard()])

❌ Violation:
class ProfileScreen extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    if (!isLoggedIn) Navigator.push(...LoginScreen...);  // imperative guard in UI
    ...
  }
}
```

---

## Deep Linking Readiness

A scalable navigation setup supports deep links without code changes. Signals:

| Signal | Meaning |
|---|---|
| Named routes via `auto_route` | Deep link ready |
| `AndroidManifest.xml` has `intent-filter` for `https` scheme | Android deep links enabled |
| `Info.plist` has `FlutterDeepLinkingEnabled: true` | iOS deep links enabled |
| Routes use only serializable params (IDs, strings) | Restorable state |
| `Navigator.push` with `MaterialPageRoute` | Deep link NOT supported for that route |

---

## Alternative: go_router

If the project uses `go_router` instead of `auto_route`, evaluate the same principles:
- All routes declared in one `GoRouter` instance
- `ShellRoute` used for bottom nav
- `redirect` callbacks used for auth guards
- Route parameters are serializable primitives

Both `auto_route` and `go_router` are acceptable. The violation is absent or scattered routing, not the choice of package.
